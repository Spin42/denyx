# Denyx in VSCode (Copilot, Continue, Cline) and Cursor

> ⚠ **Status: not thoroughly tested yet.** Denyx is exercised end-to-end with
> Claude Code and opencode (eval suite, pentest, fuzz, integration tests).
> The integrations on this page are wired via the same MCP pattern, with
> the host-specific config paths and file shapes adapted. The MCP entries
> *should* work universally (any MCP-aware host loads servers the same way);
> the host-side **lockdown** layer — what disables built-in tools so the
> model can't bypass Denyx — varies and is incomplete for some hosts. Read
> the per-host notes and treat as "should work, please file an issue when
> you confirm or break it."

## What MCP-aware host integration means

Denyx-mcp is a stdio JSON-RPC server. **Any host that loads MCP servers
can route tool calls through it.** That part is universal.

The Denyx threat model also requires *disabling* the host's built-in
effecting tools — `Read`, `Write`, `Edit`, `Bash`-equivalents, web fetch
— so the model can't bypass the policy by going around the MCP layer.
That part is host-specific. Each host below has its own answer to:

1. **Where does the host's MCP config live?** (project-local file, user
   home, extension storage, …)
2. **Does the host let you deny built-in tools project-locally?**
3. **Does the host have an OS-level sandbox you can pair with Denyx's
   gate?**

Claude Code answers all three; the rest answer some.

## Generated by `denyx host-config`

`denyx host-config` knows how to write the MCP entry for the hosts on
this page. Examples:

```sh
# Single host:
denyx host-config --policy ./denyx.toml --host cursor

# Multiple, comma-separated:
denyx host-config --policy ./denyx.toml --host claude,cursor,copilot

# Let denyx detect (env vars + cwd files), default:
denyx host-config --policy ./denyx.toml
```

Auto-detection (`--host auto`, the default) reads:

- `TERM_PROGRAM=cursor` → Cursor
- `TERM_PROGRAM=vscode` → Copilot agent mode (most common AI extension)
- `CLAUDECODE` / `CLAUDE_CODE_ENTRYPOINT` env → Claude Code
- `OPENCODE` env → opencode
- File existence in cwd: `.mcp.json`, `opencode.json`, `.cursor/mcp.json`,
  `.vscode/settings.json`, `.continue/config.json`

If detection finds nothing, host-config falls back to `claude,opencode`
with a stderr warning. Pass `--host` explicitly to override.

## Cursor

**Status:** MCP wiring is the same shape as Claude Code's `.mcp.json`,
so this is the most likely to "just work." The lockdown layer is the
weak spot.

**File written:** `<project>/.cursor/mcp.json`

**Generated content:**
```json
{
  "mcpServers": {
    "denyx": {
      "command": "denyx-mcp",
      "args": ["--policy", "./denyx.toml", "--audit-log", "./.denyx/audit.jsonl",
               "--confirm-mode", "auto"]
    }
  }
}
```

**Lockdown:** Cursor's built-in tools (Edit, Read, Bash equivalents) are
toggled in the Cursor UI under Settings → Features → Tools. There is no
project-local file Denyx can write to disable them. **You will need to
disable them manually in the UI**, and they reset per-Cursor-install — a
real ergonomics gap for unattended deployments. Until Cursor adds a
project-local deny mechanism, the Denyx gate on Cursor is *additive*
(any tool routed through `denyx_*` is gated; the model can still use
Cursor's own ungated tools alongside).

**OS sandbox:** None. Cursor doesn't ship one analogous to Claude
Code v2's bubblewrap/Seatbelt stanza.

## VSCode + GitHub Copilot agent mode

**Status:** GitHub Copilot's agent mode (GA in 2025) added MCP server
support. The exact settings key has shifted between insider builds and
stable releases — `chat.mcp.servers` is what `denyx host-config`
emits, matching the public stable shape as of early 2026. Verify against
your VSCode version's docs if the entry doesn't load.

**File written:** `<project>/.vscode/settings.json` (workspace settings)

**Generated content:**
```json
{
  "chat.mcp.servers": {
    "denyx": {
      "command": "denyx-mcp",
      "args": ["--policy", "./denyx.toml", "--audit-log", "./.denyx/audit.jsonl",
               "--confirm-mode", "auto"]
    }
  }
}
```

**Lockdown:** Copilot agent mode prompts the user before each tool
call (UI confirmation). There's no project-local deny list to block
Copilot's built-in tools the way Claude Code's `.claude/settings.json`
deny list does. The gate is therefore *additive*: Denyx-routed calls
are policy-checked; Copilot's own Read/Edit/Bash equivalents still
need user-click approval per call. For an unattended workflow, this
isn't equivalent to Claude Code's lockdown.

**OS sandbox:** None.

## VSCode + Continue

**Status:** Continue has had MCP support since 2024. The `mcpServers`
array shape in `.continue/config.json` is the supported public format.

**File written:** `<project>/.continue/config.json`

**Generated content:**
```json
{
  "mcpServers": [
    {
      "name": "denyx",
      "command": "denyx-mcp",
      "args": ["--policy", "./denyx.toml", "--audit-log", "./.denyx/audit.jsonl",
               "--confirm-mode", "auto"]
    }
  ],
  "tools": []
}
```

**Lockdown:** Continue's `tools` array is allowlist-style: any built-in
tool *not* listed is unavailable. Emitting an empty `tools: []`
disables every built-in tool, leaving only the MCP-routed `denyx_*`
calls. This is the closest of the VSCode-side hosts to a Claude
Code-style lockdown. If you want to keep specific Continue built-ins
(e.g. `readCurrentFile` for cosmetic display), add them to the array.

**OS sandbox:** None.

## VSCode + Cline / Roo Code

**Status:** Cline is the trickiest — its MCP config lives in extension
storage, not a project-local file Denyx can write to from the cwd.

**File written:** None. `denyx host-config --host cline` prints a
JSON snippet to stderr you paste into Cline's UI:

> Open Cline in VSCode → click the MCP Servers icon → Edit Global MCP
> (or Edit Project MCP if your version supports it) → paste the snippet.

**Lockdown:** Cline approves every tool call by default — UI prompts
per call. There's no project-local deny mechanism. As with Copilot,
the Denyx gate is additive rather than mandatory.

**OS sandbox:** None.

## Honest comparison

| Host | MCP wiring | Built-in deny | OS sandbox | Tested |
|---|---|---|---|---|
| **Claude Code** | `.mcp.json` | `.claude/settings.json` deny list | bubblewrap/Seatbelt stanza | ✅ |
| **opencode** | `opencode.json` | `tools` block + permission deny | — | ✅ |
| **Continue** | `.continue/config.json` | `tools` allowlist (empty = deny all) | — | ⚠ |
| **Cursor** | `.cursor/mcp.json` | UI toggles only | — | ⚠ |
| **Copilot** | `.vscode/settings.json` | per-call approval only | — | ⚠ |
| **Cline** | extension storage (manual) | per-call approval only | — | ⚠ |

**If your threat model treats the lockdown layer as load-bearing**
(must guarantee the model can't bypass Denyx), use **Claude Code**,
**opencode**, or **Continue**. Cursor / Copilot / Cline put the gate
in front of `denyx_*` calls but leave the host's built-ins available
in parallel.

**Reporting issues:** if a config below worked or didn't, please open
an issue at https://github.com/Spin42/denyx/issues with the host
version. The integration matures with feedback.
